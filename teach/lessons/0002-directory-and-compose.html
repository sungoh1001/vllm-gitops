Lesson 0001에서 서버에 Docker + NVIDIA Container Toolkit이 설치되었습니다. 이제 어디에 무엇을 둘 것인지를 결정하고, 실제로 vLLM 컨테이너를 실행해봅니다.

이 수업이 끝나면:

  • 여러분 서버의 디렉토리 구조가 결정됩니다
  • Docker Compose 파일이 무엇이고 왜 쓰는지 설명할 수 있습니다
  • vLLM 컨테이너를 GPU 0번에서 실행할 수 있습니다
---

1. 디렉토리 구조 — 왜 먼저 설계해야 하는가

핵심 원칙: Git으로 추적할 것(설정)과 Git으로 추적하지 않을 것(모델, 캐시, 시크릿)을 분리하는 것. 이것이 Git-first 워크플로우의 물리적 기반입니다.

Git 저장소 하나가 하나의 "실험 프로젝트"가 됩니다. 다음과 같은 구조를 권장합니다:

~/vllm-experiments/ ← Git 저장소 루트
├── run.sh ← vLLM 실행 명령 (실험 설정 = 이 파일)
├── compose.yaml ← Docker Compose 정의
├── .env ← HF_TOKEN 등 시크릿 (Git 추적 ❌)
├── .gitignore ← Git 제외 규칙
├── models/ ← Hugging Face 모델 캐시 (Git 추적 ❌)
├── results/ ← 실험 결과·로그 (선택적 Git 추적)
└── README.md ← 실험 노트

각 디렉토리/파일의 역할

항목Git?역할
run.sh✅실험 설정의 단일 진실 공급원. 모델명, TP 크기, 메모리 제한 등 vLLM 실행 인자를 이 파일 하나에 담습니다. git log가 곧 실험 이력이 됩니다.
compose.yaml✅컨테이너의 선언적 정의. 어떤 이미지, 어떤 포트, 어떤 볼륨, 어떤 GPU를 쓸지 명시합니다. 한 번 작성하면 잘 바뀌지 않습니다.
.env❌Git에 절대 커밋하지 않을 값들. 토큰, 비밀번호. .gitignore에 반드시 등록.
models/❌다운로드된 모델 파일. 수십 GB 단위라 Git에 넣을 수 없습니다. 컨테이너에 volume mount로 제공.
results/△실험 메트릭·로그. 작은 텍스트 파일이면 Git에 넣고, 큰 바이너리면 제외.
Lesson 0001과의 연결: Lesson 0001에서 Docker를 설치할 때 sudo usermod -aG docker $USER를 실행했습니다. 이 디렉토리는 $USER 홈 아래에 만드세요. Docker 컨테이너가 이 경로를 volume mount로 접근하게 됩니다.

2. Docker Compose — 컨테이너를 선언적으로 정의하기

Docker Compose는 "컨테이너가 이렇게 생겨야 한다"를 YAML 파일 하나에 적는 도구입니다. Lesson 0001에서 배운 docker run --gpus all ... 명령어를 기억하시나요? 그 긴 명령어를 compose.yaml 파일로 옮겨놓은 것이라 생각하면 됩니다.

GitOps 관점에서 Compose는 완벽한 짝입니다:

  • 선언적(Declarative): "GPU 0번에 이 이미지로, 이 포트로, 이 모델을 띄워라" — what을 기술
  • 버전 관리: compose.yaml이 Git에 들어가므로, 누가 언제 어떤 설정을 바꿨는지 추적 가능

vLLM용 compose.yaml — 실전 예제

# compose.yaml — Git-first vLLM 실험
services:
  vllm-dev:
    image: nexus.internal.company.com/vllm/vllm-openai:v0.7.1
    container_name: vllm-dev

    # GPU 설정
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]

    # GPU 0번만 사용
    environment:
      - NVIDIA_VISIBLE_DEVICES=0
      - HF_TOKEN=${HF_TOKEN}

    # 포트
    ports:
      - "8000:8000"

    # 볼륨 마운트
    volumes:
      - ./models:/root/.cache/huggingface
      - ./results:/workspace/results

    # 공유 메모리 (GPU 추론 필수)
    shm_size: "2gb"

    # vLLM 실행 명령 — 이 부분을 run.sh로 추출
    command: >
      --model meta-llama/Llama-3.1-8B-Instruct
      --tensor-parallel-size 1
      --gpu-memory-utilization 0.90
      --max-model-len 8192
      --dtype bfloat16
      --host 0.0.0.0
      --port 8000
      --enforce-eager

🔗 주 출처: Docker 공식 — Compose GPU 지원, vLLM 공식 — OpenAI 호환 서버

3. compose.yaml의 6가지 핵심 요소 해설

요소무엇을 결정하는가이 프로젝트에서의 선택
image 어떤 Docker 이미지를 쓸지 내부 Nexus에 미러링된 vLLM 이미지. 버전 태그(:v0.7.1)를 반드시 명시해야 재현성 확보.
deploy → devices GPU를 컨테이너에 어떻게 연결할지 driver: nvidia, count: 1로 GPU 1장 할당. Lesson 0001의 --gpus 플래그와 동일한 효과.
NVIDIA_VISIBLE_DEVICES 어떤 GPU를 쓸지 0 → GPU 0번만. 나중에 ops용으로 GPU 1번은 따로 컨테이너 띄움. 이 값만 바꾸면 GPU 할당 변경.
ports 호스트:컨테이너 포트 매핑 호스트 8000 → 컨테이너 8000. 팀원들이 http://서버IP:8000으로 API 호출 가능.
volumes 호스트 디렉토리를 컨테이너에 연결 모델 캐시(./models)와 결과(./results)를 컨테이너 내부와 공유. 컨테이너를 지워도 데이터 보존.
command 컨테이너가 실행할 명령 vLLM 서버 기동 명령. 이 부분이 실험마다 바뀌는 영역 — 그래서 run.sh로 분리함.
---

4. Volume Mount — 모델 캐시의 생명주기

이 부분을 제대로 이해하지 못하면 컨테이너를 재시작할 때마다 모델을 다시 다운로드하게 됩니다. 에어갭 환경에서는 HF Mirror에서 다시 받아야 하니 특히 중요합니다.

Volume Mount의 동작 방식

호스트:  ./models/  ────────────마운트────────────▶  컨테이너: /root/.cache/huggingface/
          (실제 물리 디스크)                              (컨테이너가 보는 가상 경로)

컨테이너 안에서 /root/.cache/huggingface/hub/에 모델 파일을 쓰면, 실제로는 호스트의 ./models/hub/에 저장됩니다. 컨테이너가 삭제되어도 호스트 디스크의 파일은 그대로 남습니다.

왜 ./models 를 Git에 넣지 않나요?
Llama-3.1-8B-Instruct 하나가 약 16GB입니다. Git은 대용량 바이너리를 다루도록 설계되지 않았습니다. 모델 파일은 컨테이너의 볼륨 마운트로 관리하고, Git에는 "어떤 모델을 썼는지"(run.sh의 MODEL 변수)만 기록합니다.
---

5. run.sh — 실험 설정의 단일 파일

compose.yaml은 인프라 정의(고정), run.sh는 실험 설정(변동)입니다. 이 분리가 Git-first 실험의 핵심입니다. run.sh 하나만 수정해서 커밋하면, 그것이 곧 하나의 실험 기록이 됩니다. 

#!/bin/bash
# run.sh — vLLM 실험 설정
# 이 파일을 수정 → git commit → git push → 자동 실행 (Lesson 0004 예정)

# === 실험 변수 (여기만 바꾸세요) ===
MODEL="meta-llama/Llama-3.1-8B-Instruct"
TP_SIZE=1
MAX_MODEL_LEN=8192
GPU_MEM_UTIL=0.90
DTYPE="bfloat16"

# === 실행 (이 아래는 바꾸지 마세요) ===
docker compose up -d

docker compose up -d 한 줄이 compose.yaml을 읽어 컨테이너를 백그라운드(-d)로 실행합니다. 컨테이너가 이미 떠 있으면 설정 변경사항을 반영해 재시작합니다.

6. vLLM 핵심 인자 — 처음 알아야 할 5개

vLLM에는 수십 개의 옵션이 있지만, 실험의 출발점은 이 5개입니다:

인자의미기본값실험 포인트
--tensor-parallel-size 모델을 몇 개의 GPU에 나눌지 1 NVLink 없으면 무조건 1. 2로 하면 PCIe 병목으로 오히려 느려짐
--gpu-memory-utilization GPU 메모리의 몇 %까지 모델에 할당할지 0.90 0.85~0.95 사이. 너무 높으면 OOM, 너무 낮으면 KV Cache 공간 부족
--max-model-len 처리할 최대 컨텍스트 길이(토큰) 모델 기본값 값이 클수록 더 많은 VRAM 사용. 작게 시작해서 점진적 증가
--dtype 추론 정밀도 "auto" A100에서는 bfloat16이 기본. float16과 속도/메모리 차이 비교해볼 만함
--enforce-eager CUDA Graph 비활성화 비활성(그래프 사용) 첫 실행 시 CUDA Graph 오류가 나면 이 플래그 추가. 안정성 ↑, 약간의 성능 ↓
TP=1이 핵심입니다. NVLink 없는 PCIe 환경에서 --tensor-parallel-size 2를 주면, GPU 간 데이터 전송이 PCIe(32GB/s)로 이루어져 단일 GPU보다 느려집니다. NVLink는 600GB/s입니다.

7. 첫 실행 — 실제 워크플로우

# 1. 디렉토리 생성
mkdir -p ~/vllm-experiments/{models,results}
cd ~/vllm-experiments

# 2. compose.yaml 작성 (위 예제 복사)
vim compose.yaml

# 3. .env 파일에 HF_TOKEN 설정 (에어갭: HF Mirror 토큰)
echo 'HF_TOKEN=your_token_here' > .env

# 4. .gitignore 생성
echo '.env' > .gitignore
echo 'models/' >> .gitignore

# 5. 컨테이너 실행
docker compose up -d

# 6. 로그 확인 (모델 로딩 1~3분 소요)
docker compose logs -f

# 7. API 테스트
curl http://localhost:8000/v1/models
에어갭 환경 특이사항: compose.yaml의 image는 Nexus 내부 주소. 모델 다운로드 경로도 HF Mirror를 바라보도록 HF_ENDPOINT 환경변수가 필요할 수 있습니다. 여러분의 Mirror 주소를 확인하세요: echo 'HF_ENDPOINT=https://hf-mirror.internal.company.com' >> .env
